# pickup_v2 串口协议调试复盘 — 2026-05-15

> 从 "MCU 一收多字节命令就死锁" 到全套 8/8 PASS 的完整复盘。
> 写给未来的自己：哪些路走对了、哪些走偏了、为什么走偏。

## 一、TL;DR（30 秒结论）

- **真正的 bug 只有两个**：
  1. **newlib-nano 的 `strtok()` 在 RA6M5 / e2studio 工具链下挂死** → 用手写 `pv2_next_token()` 替换
  2. **legacy 单字符截胡逻辑无差别拦下 `'C'`** → 把 `CLOSE\n` 的开头吞了 → 按状态过滤截胡
- **真正起作用的诊断方法只有一个**：在嫌疑函数前后插 `uart_log("[Dx]")` 打点，看 Debug COM 最后一条卡在哪
- **真正应该最先做的实验**：`Z\n`（最小用例），而不是 `HOME\n`（一次性引入 5+ 个变量）

## 二、问题表象

| 实验 | 字节 | 结果 |
|---|---|---|
| 发 `G` | 1 字节 | ✅ 通，传送带启动 |
| 发 `X` | 1 字节 | ✅ 通，急停 |
| 发 `AB` | 2 字节 | ✅ 通，走 legacy 分类 |
| 发 `HOME\n` | 5 字节 | ❌ MCU 死锁，Debug COM heartbeat 永远停止 |
| 发 `HOMX\n` | 5 字节 | ❌ 同上 |

第一时间的假设：是不是多字节场景 ringbuf / ISR / FSP FIFO 出问题？这个假设**害我们走了很多弯路**。

## 三、调试时间线（按时序）

### 阶段 0：埋下伏笔（编码时）

写 `parse_extended_cmd` 时用了 newlib 的 `strtok(line, " \t")`，理由是"标准库函数省事"。这是日后死锁的真因，但当时根本想不到。

`process_pi_char` 里写了 legacy 单字符截胡，对 `'A'/'B'/'C'/'G'/'X'` 在 `g_pi_cmd_len == 0` 时直接走 `process_pi_char_legacy`。这是日后 CLOSE bug 的真因，但当时认为是"完美的向后兼容"。

### 阶段 1：发现问题 + 错误假设

第一次实验 `HOME\n` 让 MCU 死锁。**直觉假设**：5 字节比 1 字节多 4 字节，多字节 ringbuf / ISR 有 bug。

### 阶段 2：错误方向 ① — DIAG 计数器

加了 4 个 `volatile uint32_t` 计数器（`isr / drop / take / proc`），每秒 heartbeat 打印。

**结果**：
- 单字节实验通过时计数器全对（这证明 ringbuf / ISR 工作正常）
- 多字节实验 MCU 死锁后，**heartbeat 自己也停了**——计数器读不到，等于白加

**教训**：当 MCU 主循环可能死锁时，依赖主循环打印的计数器毫无用处。只有"打点立刻输出"的方式（`uart_log`）才有效。

### 阶段 3：错误方向 ② — HOMX\n 排除法

为了"排除 HOME 命令本身的副作用"，发 `HOMX\n`（4 字节非法 + \n）。结果**也死锁**。

这步本身是对的，但**起点选错了**：HOMX\n 仍有 5 字节缓冲、`\n` 触发、`parse_extended_cmd`、`strtok` 4 个变量。**应该一开始就用 Z\n（最小用例）**，但当时没想到。

### 阶段 4：错误方向 ③ — Clean + Build

每次改完代码都让自己"Clean → Build"，怀疑增量编译没真的把改动塞进 elf。

**事后看：这条建议从头到尾都是浪费时间。** 编译没问题——后来加上 boot 指纹 `*** PICKUP_V2 XXX BUILD 2026-05-15 ***` 直接验证 elf 内含新代码，从来没出过"指纹对了但行为还是老的"的情况。Clean 加在每次烧录前最多增加 60 秒，累计浪费了大量时间。

**真正该信任的验证**：boot 指纹字符串 + ELF text size 变化。不要无差别地 Clean。

### 阶段 5：错误方向 ④ — 怀疑 FSP / FIFO / ISR

中间怀疑过：
- SCI9 `rx_fifo_trigger = MAX` 是不是等 FIFO 满才中断？→ 不是，15 ETU 就触发，已确认
- ISR 在多字节场景是不是漏字节？→ 没有，计数器 isr+2 take+2 全对
- `UART_EVENT_ERR_OVERFLOW` 之类的 ERR 事件没处理？→ callback 里没收到 ERR 事件

每条都耗时排查，每条都不是。**典型的"先入为主：底层硬件问题"**。

### 阶段 6：关键转折 — Z\n 最小用例

终于想到："把变量降到最少"。Z\n 只有 2 字节，命令本身不存在，走 NACK BADARG 路径。

**结果**：Z\n 也死锁。

这一个实验**一次性证伪了**前面所有假设：
- 字节数？不是，2 字节就死了
- 命令副作用？不是，Z 不是合法命令
- ringbuf 容量？不是，2 字节远没到 64
- 多字节 ISR？不是，AB 是 2 字节但通

嫌疑被精确锁定到 `parse_extended_cmd("Z")` 内部：`strtok` / `strcmp` 链 / `NACK BADARG` 的 `uart9_send_blocking` 三选一。

### 阶段 7：正确方向 — uart_log 打点

在 `parse_extended_cmd` 内插 6 个 `uart_log("[Dx]")`，在 `process_pi_char` 的 `\n` 分支插 2 个 `uart_log("[Nx]")`。

`uart_log` 走独立的 SCI7 + 200000 次 busy-wait timeout 保护，**最多卡 5ms 就强退**，不会被嫌疑函数拖累。

第一轮跑 Z\n：
```
[N1]
[D1]
```
然后停。**死在 `[D1]` 之后到 `[D2]` 之前**。`[D1]` 后只有 `char * tok = strtok(line, " \t");` 一行。

为了排除"`uart_log` 自己没在 [D1] 后正确返回"的可能，加了手写 hex dump（不用 snprintf，零库依赖）：

```
[N1]
[D1]
BUF=5A000000   ← 'Z' + 三个 \0，buffer 内容完全正常
[D1b]
```

**确诊**：strtok(line, " \t") 对一个 "Z\0\0\0" 的合法 buffer + " \t" delim 直接死锁整个 main loop。

### 阶段 8：修复 ① — 替换 strtok

写一个 `pv2_next_token(char ** p_save)`，行为类似 `strtok_r` 但**零外部依赖**（不调任何库函数，不依赖任何 static state）。把 6 处 `strtok` 调用全部换掉，cmd_handler 签名改为接收 `char **`。

Z\n 立刻通过，Debug COM 完整打出 `[N1] [D1] BUF=5A000000 [D1b] [D2] [D3] [D4] [D5] [D6] [N2]`，Pi 端收到 `b'NACK BADARG\n'`。

### 阶段 9：次生 bug — CLOSE\n 被吞

跑全套测试时，OPEN/HOME/K 都通，但 CLOSE 失败：返回 `NACK BADARG` 而不是 `READY`。

**根因**：legacy 截胡逻辑在 `g_pi_cmd_len == 0` 时无差别拦下 `'C'`。CLOSE 的第一个 'C' 被当作"分类未熟草莓"走 legacy，剩下的 "LOSE\n" 进缓冲 → parse_extended_cmd("LOSE") → 不匹配 → NACK BADARG。

这是**协议设计层面的冲突**：'C' 既是 legacy 单字符，又是 "CLOSE" 字头。

### 阶段 10：修复 ② — legacy 按状态过滤

把截胡条件从"字符在 ABCGX 表里"改为"该字符在当前状态下作为 legacy 有意义"：

| 字符 | 触发 legacy 的状态 |
|---|---|
| `X` | 任意（急停永远优先） |
| `G` | 仅 `STATE_IDLE` |
| `A`/`B`/`C` | 仅 `STATE_BELT_RUN`（分类设值场景） |

其他情况让字符正常进入行缓冲，由 `\n` 触发 `parse_extended_cmd`。CLOSE 立刻通过，全套 8/8 PASS。

### 阶段 11：清理

删掉所有 DIAG 相关代码（4 个计数器、`[Dx]/[Nx]` 探针、hex dump、heartbeat printf）。代码恢复整洁。

## 四、为什么会遇到这些问题

### Q1: 为什么 strtok 会挂死？

newlib-nano 的 `strtok` 用 static state 保存切分位置，可能依赖 `_REENT` / `_impure_ptr` 结构。在 RA6M5 + e2studio 工具链默认配置下，这个结构可能没正确初始化，导致 strtok 内部访问非法地址 → CPU 触发 HardFault → FSP 默认 `HardFault_Handler` 是 `while(1);` → main loop 永不回来。

**这是嵌入式 C 的常见坑**。strtok / sprintf / sscanf / malloc 这些依赖 newlib 状态的函数在嵌入式环境都不完全可靠，**MISRA-C 等规范明确禁用 strtok**。

### Q2: 为什么 CLOSE 被吞？

协议设计时**没有意识到**：

1. legacy 单字符 ABCGX 是"立即触发"语义
2. 扩展命令是"\n 触发"语义
3. 这两套语义共用同一份 process_pi_char 字节流

只要扩展命令字头跟 legacy 单字符有任何一个字符冲突（这次是 'C'），就会出 bug。当时设计选 CLOSE 这个英文单词时根本没意识到与 'C' 冲突。

### Q3: 为什么调试时走了那么多弯路？

3 个深层原因：
1. **先入为主**：看到"多字节失败、单字节成功"立刻假设是"字节数"问题，没考虑"`\n` 触发的 parse 路径"
2. **没用最小用例**：第一个实验就用 HOME\n（5+ 个变量），应该用 Z\n（1 个变量）
3. **依赖了不可靠的工具**：DIAG 计数器在死锁场景失效；后来才换上 uart_log 打点

## 五、走过的弯路（防再犯清单）

1. ❌ **盲目 Clean + Build** — 在没有"指纹不对"或"size 不变"等证据时，Clean 是浪费时间。信任 boot 指纹和 ELF size 变化即可。
2. ❌ **DIAG 计数器排查死锁** — 计数器靠 main loop 打印，main loop 死了就读不到。用 `uart_log` 立即打点替代。
3. ❌ **怀疑 FSP / FIFO / ISR** — 这些是低概率层，在没有强证据时不要先怀疑。先用最小用例锁定到应用层代码，再向下挖。
4. ❌ **从大用例开始排查** — 多变量场景排错从最小用例开始（这次是 Z\n，而不是 HOME\n）。
5. ❌ **用标准库字符串函数** — strtok / sscanf / sprintf 在嵌入式 newlib 下都列入嫌疑名单。手写小函数比用库更可控。

## 六、调试方法论沉淀（下次直接套用）

### MCU 死锁定位三步法

1. **加 boot 指纹**：`uart_log("[BOOT] *** XXX BUILD YYYY-MM-DD ***\r\n")`，每次烧录前改一次。烧录后看到指纹就证明 elf 内含本次改动。
2. **加最小用例**：把命令简化到"最少字节 + 最少状态 + 最少副作用"。这次是 `Z\n`，2 字节、走 NACK 路径、不动舵机。
3. **加 uart_log 打点**：在嫌疑代码段每个分支前后插 `uart_log("[Dx]")`，看 Debug COM 最后一条停在哪。`uart_log` 必须走独立 SCI 通道 + busy-wait timeout，确保它自己不会被嫌疑函数拖累。

### 嵌入式 C 字符串处理黑名单

| 函数 | 替代 |
|---|---|
| `strtok` / `strtok_r` | 手写 `next_token(char **p_save)` |
| `sprintf` | 手写 hex / dec 编码（这次用了零库依赖的 HEX 数组方案） |
| `sscanf` | 手写字段切分 + 简单 atoi / strtof |
| `malloc` / `free` | 静态 buffer / 栈缓冲 |

### 协议设计审查清单

设计任何命令字时**先做一遍冲突检查**：

- [ ] 扩展命令字头不能与 legacy 单字符冲突（这次 C ↔ CLOSE 翻车）
- [ ] 截胡逻辑要按"该字符在当前上下文是否有意义"过滤
- [ ] 终结符（`\n`）路径和立即触发路径不要混用，或者明确两者的优先级

## 八、不会再犯的清单（贴墙上）

1. **任何"MCU 收命令死锁"问题，第一个实验永远是 `Z\n`，不是真实命令**
2. **任何嫌疑函数前后都先加 `uart_log` 打点，不要先去怀疑 FSP / ISR / FIFO**
3. **嵌入式不用 strtok / sscanf / sprintf；要切字符串就手写 next_token**
4. **协议设计时把所有 legacy 单字符列出来，跟所有扩展命令字头做交叉检查**
5. **Clean + Build 不是万能药；信任 boot 指纹和 ELF size，不是"以防万一"地 Clean**
6. **多字节协议必须考虑：每个字节到达时，当前缓冲状态是什么？\n 触发什么？立即触发什么？两者会不会打架？**

**调试用时：跨 2 个 session 共数小时。如果一开始就直接 Z\n + uart_log 打点，应该 30 分钟内能定位。**
